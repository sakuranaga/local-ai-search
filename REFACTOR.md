# リファクタリング候補一覧

## 優先度: 中

### 1. エラーハンドリングの統一
**場所**: バックエンド全体

各ルーターで個別にHTTPException を投げているが、エラーレスポンスの形式が不統一。

- 共通のエラーレスポンススキーマを定義
- カスタム例外ハンドラーで統一的に処理

### 2. パーミッションチェックの依存性注入化
**場所**: `backend/app/routers/documents.py`

`_check_doc_access()`, `_check_folder_access()` が関数内で直接呼ばれている。

- FastAPI の `Depends()` パターンで依存性注入に変更
- ルート定義時点でパーミッション要件を明示

## 優先度: 低

### 3. CSS クラスの整理
**場所**: `frontend/src/pages/FileExplorerPage.tsx`

Tailwind クラスが非常に長い文字列になっている箇所が多数。

- `cn()` ユーティリティの活用
- 繰り返しパターンを `@apply` でカスタムクラス化

### 4. マジックナンバーの定数化
**場所**: 複数ファイル

- チャンクサイズ（`1000`, `200`）
- ページサイズ（`30`, `50`, `200`）
- タイムアウト値
- ファイルサイズ制限

→ `backend/app/constants.py`, `frontend/src/lib/constants.ts` に集約

### 5. テストの追加
**場所**: プロジェクト全体

現在テストがほぼ存在しない。優先的にテストすべき箇所:

- パーミッションチェックロジック
- 検索サービス（フィルタ組み合わせ）
- ドキュメント処理パイプライン
- API キー認証フロー

### 6. DB マイグレーションの整理
**場所**: `backend/alembic/versions/`

マイグレーションファイルが増加傾向。スキーマが安定したタイミングでスカッシュを検討。

### 7. 型安全性の強化
**場所**: フロントエンド全体

- `any` 型の使用箇所を削減
- API レスポンスの型ガード追加
- `zod` などによるランタイムバリデーション検討

### 8. ログの構造化
**場所**: バックエンド全体

現在 `logger.info()` / `logger.error()` で非構造化テキスト。

- 構造化ログ（JSON形式）への移行
- リクエストID のトレーシング

### 9. 設定管理の見直し
**場所**: `backend/app/config.py`

環境変数とDB設定（`SystemSetting`）が混在。

- どの設定がどこに保存されるかのドキュメント整理
- 起動時バリデーションの強化

### 10. フォルダ操作のバッチ化
**場所**: `backend/app/routers/documents.py`

フォルダ削除時に子ドキュメントを1件ずつ処理。大量データ時にN+1問題。

- バルクUPDATE/DELETE クエリに変更

### 11. ChatPanel のストリーミング状態管理
**場所**: `frontend/src/components/ChatPanel.tsx`

`useRef` と `useState` の混在で状態管理が複雑。

- `useReducer` パターンでストリーミング状態を一元管理
- `accumulated` 変数のクロージャ依存を解消

## 権限モデル調査結果

### 管理者だけができること（APIレベルで403）

| 機能 | エンドポイント |
|------|------------|
| ユーザー管理 (CRUD) | `/users/*` |
| ロール管理 | `/roles/*` |
| システム設定 (LLM URL等) | `/settings/*` |
| API キー管理 | `/api-keys/*` |
| グループ作成・削除・メンバー操作 | `/groups/*` (一覧以外) |

### ドキュメント・フォルダレベルの違い

| 操作 | 一般ユーザー | 管理者 |
|------|-----------|--------|
| 一覧表示 | 自分のもの + others_read + グループ共有のみ | 全ドキュメント |
| 読み取り | owner / group_read / others_read | 常にOK |
| 書き込み | owner / group_write / others_write | 常にOK |
| 権限変更 | 自分が owner のみ | 全ドキュメント |
| ゴミ箱 | 自分のもの only | 全員のゴミ箱を操作可能 |
| フォルダ一覧 | 自分のもの + 共有のみ | 全フォルダ |

### Unixスタイルパーミッション（ドキュメント毎）

```
rw  r-  r-
│   │   └── others (全員)
│   └────── group (所属グループ)
└────────── owner (常にrw固定)
```

評価の優先順位: `admin > owner > group > others`
- グループメンバーの場合、group の設定が適用され others にフォールバック**しない**

### フロントエンド

- 管理画面のリンクは `roles.includes("admin")` で表示/非表示
- ルートガードはなし（直接 `/admin` にアクセス可能だが、APIが403を返す）

### セキュリティ上の問題点

#### 要修正（権限チェック欠如）

| 問題 | 影響 | 推奨対応 |
|------|------|---------|
| タグの作成・編集・削除に権限チェックなし | 一般ユーザーが他人のタグを削除可能 | admin only にする |
| フォルダの更新（リネーム等）に権限チェックなし | 一般ユーザーが他人のフォルダをリネーム可能 | owner or admin only |
| フォルダ削除に権限チェックなし | 一般ユーザーが他人のフォルダを削除可能 | owner or admin only |
| フォルダ権限変更が owner_id 変更時のみチェック | group_id 等の変更はチェックなし | 全権限変更を owner or admin only に |
| フロントエンドに `/admin` ルートガードなし | エラー画面が一瞬見える | 軽微、後回し可 |

#### 現状で適切な点

- ドキュメントのUNIXパーミッション（owner/group/others）は正しく実装
- `admin > owner > group > others` の優先順位は正しい
- ゴミ箱は owner or admin のみ
- フォルダ一覧は `build_folder_visibility_filter` で正しくフィルタ済み
- フォルダ作成は全員OK（`owner_id=current_user.id` で自動設定、妥当）
- デフォルト権限 `rw--r-` は安全側
- ドキュメント件数もドキュメント可視性フィルタを通して計算済み
