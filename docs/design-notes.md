# ノート / Wiki 機能 — 詳細設計

## 概要

既存ドキュメントにBlockNoteエディタによるリッチノート機能を付加する。ノートはドキュメントの一種であり、ファイルとしても存在し続ける。保存時にMarkdownに変換して検索テキストに書き込むことで、検索・AI参照の対象になる。

## 設計思想

- **ノートはファイルへの付加機能** — 独立した別物ではない
- すべてのドキュメント（PDF、動画、画像等）にノートを付けられる
- ノートを付けたファイルはフォルダツリーから消えない（📝マークが付くだけ）
- ノートツリーはフォルダ階層とは独立した階層構造を持つ
- BlockNote → Markdown → `content`（検索テキスト）に変換し、既存の検索インフラに乗る
- バージョン管理・パーミッション・共有・タグは既存機能をそのまま利用

## データモデル

既存の `documents` テーブルに `is_note` フラグを追加。ノートもドキュメントの一種として統一管理する。

```sql
ALTER TABLE documents ADD COLUMN IF NOT EXISTS is_note BOOLEAN DEFAULT FALSE;
ALTER TABLE documents ADD COLUMN IF NOT EXISTS note_content JSONB;        -- BlockNote のブロックデータ
ALTER TABLE documents ADD COLUMN IF NOT EXISTS parent_note_id UUID REFERENCES documents(id) ON DELETE SET NULL;
ALTER TABLE documents ADD COLUMN IF NOT EXISTS note_order INTEGER DEFAULT 0;  -- 同階層内の並び順
CREATE INDEX IF NOT EXISTS ix_documents_parent_note_id ON documents(parent_note_id);
```

- `is_note = true` のドキュメントはノート機能が有効
- `note_content` に BlockNote のブロック構造を JSON で保存（エディタ状態）
- `parent_note_id` でノート独自の親子階層を構成（フォルダとは独立）
- `note_order` で同階層内の表示順を制御（ドラッグ&ドロップ並び替え）
- `parent_note_id = NULL` のノートがトップレベル
- **フォルダツリーにもノートは表示される**（📝マーク付き）。フォルダから消えない
- ノートツリーとフォルダツリーは独立（互いに干渉しない）

### 保存フロー

```
BlockNote 編集 → 保存
  ├→ note_content (JSONB) … エディタのブロック状態を保存
  └→ content (TEXT) … Markdown に変換して検索テキストに上書き
       ├→ チャンク分割 → embedding → ベクトル検索
       └→ pg_bigm → 全文検索
```

- 保存時に BlockNote JSON → Markdown 変換 → `content` に書き込み
- `content` が更新されるので、既存のチャンク分割・embedding 生成が自動で動く
- バージョン管理も `change_type="text_edit"` で自動記録

## UI 設計

### ノート作成フロー（Notion 方式）

1. サイドバー「ノート」の **＋ボタン** クリック
2. 「無題のノート」で**即作成**（モーダルなし、タイトル入力なし）
3. 右ペインに BlockNote エディタが開く
4. タイトルはエディタ上部で編集可能
5. すぐ書き始められる

**ファイルアップロード（＋ボタン）は既存のまま残す。** ファイル管理とノート作成は別操作。

### サイドバー

```
[＋ アップロード]              ← 既存のまま（ファイル追加）

📁 フォルダ
  📁 営業資料
    見積書_A社.pdf
    プロジェクト計画書.md 📝   ← ノートマーク付き（フォルダから消えない）
  📁 議事録
    議事録_03-21.md 📝
    会議動画.mp4 📝            ← 動画にもノートを付けられる
📌 お気に入り
🏷️ タグ
📝 ノート  [＋]                ← ノート新規作成（即作成）
  📝 プロジェクトA              ← parent_note_id = NULL（トップレベル）
    📝 設計メモ                 ← parent_note_id = プロジェクトA
    📝 議事録 03-21
      📝 補足メモ               ← 無制限ネスト
  📝 個人メモ
    📝 TODO
  📝 無題のノート
🗑️ ゴミ箱
```

- ノートの階層はフォルダとは完全に独立（2つの階層は互いに干渉しない）
- ファイルをフォルダ移動してもノートの親子関係は変わらない
- ノートの階層を変えてもファイルの所属フォルダは変わらない
- フォルダツリーではノートファイルに📝マークが表示される
- 右クリックで「サブノート作成」→ 親ノートの下に子ノートを追加
- ドラッグ&ドロップで階層移動・並び替え（`parent_note_id` + `note_order` 更新）

### 既存ファイルへのノート追加

ファイル右クリックメニューに「ノートに追加」を追加。

**フロー:**

1. 右クリック → 「ノートに追加」
2. バックエンドで `is_note = true` に更新
3. 既存の `content`（検索テキスト）を BlockNote JSON に変換して `note_content` に格納
4. フォルダツリーでファイル名に📝マークが表示される
5. サイドバーのノートツリーにも表示される
6. ノートツリーからクリックするとBlockNoteエディタで編集可能

**テキスト → BlockNote JSON 変換ルール:**

- 検索テキストを行ごとに分割し、各行を paragraph ブロックに変換
- マークダウン記法（`#`, `-`, `>`, ``` 等）が含まれている場合は対応ブロックタイプに変換
- ファイルは消えない。ノートが付加されるだけ

### 新規ノート作成

1. サイドバー「ノート」の＋ボタンクリック
2. バックエンドで空の MD ファイルを作成（`documents` レコード + 物理ファイル）
3. `is_note = true`、タイトル「無題のノート」で即作成
4. 右ペインにBlockNoteエディタが開く

## AI コマンド設計

### アーキテクチャ

```
BlockNote スラッシュメニュー
  ↓ /ai 要約
フロントエンド
  ↓ POST /api/chat (streaming)
バックエンド
  ↓
Ollama (ローカル LLM)
```

### スラッシュメニュー AI コマンド

| コマンド | 説明 | プロンプト概要 |
|----------|------|---------------|
| `/ai 要約` | 選択テキストを要約 | 「以下を簡潔に要約してください」 |
| `/ai 翻訳` | 選択テキストを翻訳（日↔英） | 「以下を翻訳してください」 |
| `/ai 補完` | カーソル位置の続きを生成 | 「以下の続きを書いてください」 |
| `/ai 校正` | 誤字脱字・文法チェック | 「以下の文章を校正してください」 |
| `/ai 箇条書き` | 長文を箇条書きに変換 | 「以下を箇条書きに変換」 |

### UI フロー（xl-ai 風）

1. テキスト選択 or カーソル位置で `/ai` 入力
2. サブメニューから操作を選択
3. AI がストリーミングでゴーストブロック（薄い色）に出力
4. **Accept** / **Reject** ボタンで確定 or 破棄
5. Accept → ゴーストブロックが通常ブロックに変換

## リアルタイム共同編集（Yjs）

### アーキテクチャ

```
┌──────────┐   WebSocket   ┌─────────────────┐   永続化    ┌────────────┐
│ ユーザーA │──────────────►│                 │───────────►│            │
│ BlockNote │◄──────────────│  y-websocket    │            │  LevelDB   │
└──────────┘               │  (Node.js)      │            │  (永続化)   │
                           │  :1234          │            └────────────┘
┌──────────┐   WebSocket   │                 │
│ ユーザーB │──────────────►│                 │   on update  ┌────────────┐
│ BlockNote │◄──────────────│                 │─────────────►│  FastAPI   │
└──────────┘               └─────────────────┘  (callback)  │  → DB保存  │
                                                            └────────────┘
```

### y-websocket サーバー

Docker コンテナとして追加。ドキュメントごとに1つの Yjs ルームを作成。

```yaml
# docker-compose.yml
y-websocket:
  image: node:22-alpine
  working_dir: /app
  command: npx y-websocket --port 1234 --host 0.0.0.0
  ports:
    - "1234:1234"
  environment:
    YPERSISTENCE: ./yjs-data
  volumes:
    - yjs-data:/app/yjs-data
```

### ルーム管理

- ルーム名 = `note:{document_id}`（例: `note:550e8400-e29b...`）
- ユーザーがノートを開くと WebSocket 接続 → ルームに参加
- 閉じると自動退出
- y-websocket が差分同期を自動処理（CRDT）

### フロントエンド接続

```typescript
import * as Y from "yjs";
import { WebsocketProvider } from "y-websocket";
import { useCreateBlockNote } from "@blocknote/react";

const doc = new Y.Doc();
const provider = new WebsocketProvider(
  "ws://localhost:1234",
  `note:${noteId}`,
  doc
);

const editor = useCreateBlockNote({
  collaboration: {
    provider,
    fragment: doc.getXmlFragment("blocknote"),
    user: { name: currentUser.display_name, color: "#ff0000" },
  },
});
```

### カーソル・プレゼンス表示

- Yjs awareness protocol で各ユーザーのカーソル位置・選択範囲をリアルタイム共有
- ユーザー名 + カラーラベルがカーソル横に表示される（Google Docs 風）
- BlockNote の `collaboration.user` に `name` と `color` を渡すだけで自動表示

### DB への永続化

y-websocket の LevelDB が一次永続化。加えて定期的に DB に同期：

1. **保存トリガー**: デバウンス（2秒無操作後）でフロントから `PATCH /api/notes/{id}` を呼ぶ
2. **リクエスト内容**: Yjs ドキュメントから BlockNote JSON を取得して送信
3. **バックエンド処理**:
   - `note_content`（JSONB）を更新
   - BlockNote JSON → Markdown 変換 → `content`（検索テキスト）に書き込み
   - チャンク再分割 → embedding 再生成 → 検索インデックス更新
   - バージョン記録（`change_type="text_edit"`）

### 同時編集の範囲

| 対象 | 同時編集 | 方式 |
|------|----------|------|
| ノート（BlockNote） | リアルタイム共同編集 | Yjs + y-websocket |
| 検索テキスト（OverType） | 早い者勝ち（last-write-wins） | 既存のまま変更なし |
| ファイル上書きアップロード | 早い者勝ち | 既存のまま変更なし |
| メタ情報編集 | 早い者勝ち | 既存のまま変更なし |

## 仕様決定事項

### 1. ノート保存時の検索テキスト

- BlockNote → Markdown → `content`（検索テキスト）を**丸ごと上書き**
- 元のOCRテキスト等は消えるが、バージョン履歴で復元可能
- `is_note = true` のとき、詳細モーダルの「検索テキスト」タブは**非表示**（ノートが検索テキストを兼ねるため不要）
- ノートの編集はノートツリー → 右ペインの BlockNote エディタから行う

### 2. ノート有効ファイルの上書きアップロード

- ファイル上書き時は通常通りテキスト抽出 → `content` 更新（既存フロー変更なし）
- `note_content`（JSONB）はそのまま残る
- 次にノートを開いて保存すれば、ノートMDで `content` が再度上書きされる
- 特別な処理は不要

### 3. ノート解除

- 右クリック or 詳細モーダルから「ノートを解除」可能
- `is_note = false`, `note_content = NULL`, `parent_note_id = NULL` にリセット
- フォルダツリーの📝マークが消え、ノートツリーからも消える
- ファイルと検索テキストはそのまま残る

### 4. ノート削除

ノートツリーで削除時、確認ダイアログで選択肢を表示：

```
┌─ ノートを削除 ──────────────────┐
│                                  │
│  「プロジェクト計画書」のノートを  │
│  削除します。                     │
│                                  │
│  ○ ノートのみ削除                │
│    （ファイルは残ります）          │
│                                  │
│  ○ ファイルごと削除              │
│    （ゴミ箱に移動します）          │
│                                  │
│         [キャンセル] [削除]       │
└──────────────────────────────────┘
```

- 「ノートのみ削除」→ ノート解除（`is_note = false`）
- 「ファイルごと削除」→ ゴミ箱行き（`deleted_at` セット）

### 5. 親がいなくなった時の子ノート（共通ルール）

親ノートが削除（ノート解除 or ファイル削除）された場合：
- 子ノートは**トップレベルに移動**（`parent_note_id = NULL`）
- 子ノート自体は影響を受けない（ファイルもノート状態も維持）
- ノート削除・ファイル削除どちらのケースも同じ動作

### 5a. ファイル削除時のノートの扱い

- ファイルをゴミ箱に入れた場合、ノートもゴミ箱行き（ノートツリーから消える）
- ファイル復元でノートも復元される
- 子ノートは上記共通ルールでトップレベルに移動

### 5b. 管理機能

- 管理画面にノート一覧を表示
- 一括ノート解除・一括削除が可能

### 6. ノートの権限

- 既存のドキュメント権限（owner/group/others）をそのまま利用
- 新規ノート作成時は `folder_id = NULL`、デフォルト権限（`others_read = true`）

### 7. フォルダツリーで📝ファイルをクリック

- 通常のファイル詳細モーダルが開く（既存動作のまま）
- ノートの編集はノートツリー側から行う

### 8. データの正（source of truth）

- **PostgreSQL が常に正**
- y-websocket の LevelDB はリアルタイム同期用キャッシュ
- ページを開き直したら DB から `note_content` を読み込んで Yjs ドキュメントを再構築
