# ファイルとコンテンツの設計思想

## コンセプト

LASは**ファイルサーバー**である。元ファイルは不可侵が原則。
変更はアップロードによる上書き（overwrite）のみ許可する。

メモ、要約、権限、タグなどの付随データは**メタデータ**として扱い、元ファイルを壊さない。

## データの流れ

```
元ファイル(source_path) → テキスト抽出 → content(DB) → チャンク → 検索インデックス
```

- `source_path`: ディスク上の元ファイル。アップロードされたまま保持。
- `content`: 検索用に抽出されたテキスト。DBカラム。
- `note_content`: BlockNote JSON。ノート機能用。DBカラム。
- チャンク: `content`を分割してembedding付きで保存。全文検索・ベクトル検索に使用。

## 検索テキスト編集

OCR失敗や動画など抽出不可ファイルのために、`content`を手動編集できるようにした。
これはあくまで**検索用テキストの補正**であり、元ファイルの編集ではない。

## ノート機能

`content`フィールドを活用し、ノート機能を追加した。

- ノート新規作成: 空の`.md`ファイルを生成し、`note_content`(BlockNote JSON)で編集
- 既存ファイルのノート化: `.md`ファイルのみ対象。`content`をBlockNote JSONに変換
- ノート保存時: `note_content` → markdown変換 → `content`に保存 → チャンク再生成

## .mdファイルの特別扱い

ここで矛盾が発生した:

| ケース | 元ファイル | content(DB) | 問題 |
|--------|-----------|-------------|------|
| PDF/動画を検索テキスト編集 | 変わらない | 変わる | なし（設計通り） |
| 新規ノート(.md) | 0バイトのまま | 変わる | ファイルが永久に空 |
| 既存.mdをノート化して編集 | 変わらない | 変わる | ダウンロードすると古い内容 |
| .mdを検索テキスト編集 | 変わらない | 変わる | ダウンロードすると古い内容 |

### 決定事項

**`.md`ファイルのみ、content変更時に元ファイルへ書き戻す。**

理由:
- ノートの実体が`.md`。ノート新規作成も、ノート化できるのも`.md`のみ。
- `.md`は`content ≒ ファイル内容そのもの`。書き戻さないとダウンロード時に不整合。
- `.txt`等は対象外。ログファイルやCSVが`.txt`で上がる可能性があり、検索テキスト編集での書き換えは「元ファイル不可侵」の思想に反する。
- `.md`＝編集可能なドキュメント、それ以外＝閲覧専用、というシンプルなルール。

### 適用箇所

1. **ノート保存**(`notes.py: _update_search_index`): note_content → markdown → content → 元ファイル書き戻し
2. **検索テキスト編集**(`documents.py: update_document`): content変更 → 元ファイル書き戻し

両方のパスで統一的に適用する。`file_size`も更新する。
