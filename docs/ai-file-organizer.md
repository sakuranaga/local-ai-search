# AI ファイル振分けエージェント 設計書

## 概要

未整理（フォルダ未設定）のファイルに対して、AI がファイル内容を読み取り、
最適なファイル名・フォルダ・タグを提案または自動適用する機能。

2つのモード:
- **提案モード**: AI が推奨を表示し、ユーザーがクリックで適用
- **自動モード**: AI が自動的にファイル名変更・フォルダ移動・タグ付与

## 対象

**未整理ファイルのみ**（`folder_id IS NULL`）。
既にフォルダに入っているファイルは対象外。

## AI エージェントの動作

### ツール

既存の検索エージェントと同様の ReAct 方式。以下のツールを使用:

| ツール | 説明 |
|--------|------|
| `read_document` | ファイルの内容を読む（パーミッション無視、システム権限で実行） |
| `list_folders` | フォルダ一覧（階層構造 + 目的）を取得 |
| `list_folder_contents` | 指定フォルダ内のファイル一覧を取得 |
| `list_tags` | タグ一覧を取得 |
| `get_file_metadata` | ファイルのメタ情報（タイプ、サイズ、作成日、Exif等） |

### 判断フロー

```
1. ファイルの内容を読む (read_document)
2. メタ情報を取得 (get_file_metadata)
3. 現在のフォルダ構成を把握 (list_folders)
   → 各フォルダの「目的」と既存ファイルを確認
4. 現在のタグ一覧を把握 (list_tags)
5. 振分けルール（ユーザー定義）を参照
6. 以下を判断:
   - 推奨ファイル名（1〜3案）
   - 推奨フォルダ（1〜3案、既存 or 新規作成）
   - 推奨タグ（1〜3案、既存 or 新規作成）
7. 判断理由を記録
```

### システムプロンプト

```
あなたはファイル管理アシスタントです。
未整理のファイルを最適なフォルダに分類し、適切なファイル名とタグを提案してください。

## 利用可能なツール
- read_document: ファイルの内容を読む
- list_folders: フォルダ一覧（階層構造と目的）を取得
- list_folder_contents: フォルダ内のファイル一覧を取得
- list_tags: タグ一覧を取得
- get_file_metadata: ファイルのメタ情報を取得

## 分類の原則
1. フォルダは「誰が」「何のために」使うかで分ける（部門・プロジェクト単位）
2. フォルダは3階層以内に収める（深すぎると見つからない）
3. 文書タイプ（議事録、見積書等）はタグで管理する（フォルダではなく）
4. 1つのファイルは1つのフォルダにしか入れないが、タグは複数付けられる
5. 迷ったら上位の広いフォルダに入れる（細かすぎる分類は避ける）

## 判断基準
- フォルダの「目的」に最も合致するフォルダを選ぶ
- フォルダ内の既存ファイルも参考にする（似た内容のファイルが多いフォルダを優先）
- 適切なフォルダがない場合は新しいフォルダ名と目的を提案（ただし3階層以内）
- ファイル名は内容を端的に表す名前にする
- タグは内容のカテゴリ・キーワードを選ぶ（文書タイプの横断分類にはタグを使う）

## ユーザー定義の振分けルール
{rules}

## 回答形式（JSON）
{
  "suggestions": [
    {
      "filename": "推奨ファイル名",
      "folder": "フォルダ名（パス）",
      "folder_description": "新規フォルダの場合の目的（既存フォルダなら省略）",
      "tags": ["タグ1", "タグ2"],
      "reason": "判断理由"
    }
  ]
}

提案は1〜3案を目処に。最も自信のある案を先頭に。
```

## フォルダの「目的」フィールド

### データモデル

Folder テーブルに追加:

```sql
ALTER TABLE folders ADD COLUMN description TEXT DEFAULT '';
```

### 動作
- ユーザーが手動でフォルダを作成 → 任意で目的を記述
- AI がフォルダを作成 → 必ず目的を記述
- フォルダ作成・編集ダイアログに「目的」テキストエリア追加

## フォルダ階層の扱い

### 階層制限

フォルダの最大ネスト深度は **5階層** とする（システム制限）。
AI が新規フォルダを作成する際は **3階層以内** に制限する（分類の原則）。
ユーザーは手動で最大5階層まで作成可能。

### サブフォルダの目的の解釈

サブフォルダの「目的」は、**親フォルダの目的を継承**して解釈する。
AI はファイルの分類時に、ルートからそのフォルダまでの全階層の目的を連結して判断する。

例:
```
営業部/           目的: "営業部門の文書全般"
  └── マニュアル/  目的: "マニュアルを保存する"
```
→ AI が認識する実質的な目的: **「営業部門のマニュアル」**

同じ名前・同じ目的のサブフォルダでも、親の文脈が異なれば別の分類先として正しく区別される。

### LLM に渡すフォルダ構成

```
## 現在のフォルダ構成
- 営業部/ — 営業部門の文書全般
  - マニュアル/ — 営業部門のマニュアル
  - 契約書/ — クライアントとの契約書類
- システム部/ — システム部門の文書全般
  - マニュアル/ — システム部門のマニュアル
  - サーバー管理/ — サーバーの設定・手順・ログ
```

## 振分けルール（ユーザー定義）

### データモデル

```sql
CREATE TABLE organizer_rules (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    rule_text TEXT NOT NULL,          -- 自然言語のルール
    priority INTEGER DEFAULT 0,       -- 優先度（高い順に適用）
    is_active BOOLEAN DEFAULT TRUE,
    created_by_id UUID REFERENCES users(id),
    created_at TIMESTAMPTZ DEFAULT now()
);
```

### ルール例

```
「ファイルの中身が議事録の場合、ファイル名に登録日もしくはファイル内に書いてある
 実施日をyyyy年mm月dd日形式で頭に付けること」

「画像の場合はOCRされたテキスト以外に、画像の内容も把握し、
 Exif情報もあればそれを加味して判断してください」

「見積書・請求書は営業部/経理フォルダに分類し、
 タグに取引先名を付けてください」

「英語のドキュメントには"英語"タグを付けてください」
```

### ルールの適用

エージェントのシステムプロンプトにルールを全て注入:

```
## ユーザー定義の振分けルール
1. ファイルの中身が議事録の場合、ファイル名に...
2. 画像の場合はOCRされたテキスト以外に...
3. 見積書・請求書は営業部/経理フォルダに...
```

## LLM リソース管理

### 優先度

```
検索/チャット（最優先） > AI振分け（最低優先）
```

- 検索/チャットリクエスト中は AI 振分けを一時停止
- アイドル時に AI 振分けを実行

### エージェントループ

エージェントはイベント駆動ではなく、**ポーリング型**で動作する。
常に未整理フォルダと処理状況を監視し、アイドル時に1ファイルずつ処理する。

```
ループ:
  1. LLM が使用中か確認
     ├ 使用中（検索/チャット中） → スリープして再チェック
     └ アイドル → 2へ

  2. 未整理ファイルから処理対象を探す
     - 提案がまだないファイル
     - 提案済みだがファイルが更新されたファイル（updated_at > 提案日時）
     ├ 対象なし → スリープして1へ
     └ 対象あり → 3へ

  3. 1ファイルを処理
     - ファイル内容を読む
     - フォルダ・タグ一覧を取得
     - 振分けルールを読み込み
     - LLM に分類リクエスト
     ├ 提案モード → 提案を organizer_proposals に保存
     └ 自動モード → 即座に適用（名前変更・移動・タグ付与）+ ログ記録

  4. 1へ戻る
```

処理中に検索/チャットが来たら、現在の1ファイル処理を完了させてから一時停止。

## エージェント状態管理

### データモデル

```sql
CREATE TABLE organizer_proposals (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    suggestions JSONB NOT NULL,       -- AI の提案（1〜3案）
    status TEXT DEFAULT 'pending',    -- 'pending' | 'applied' | 'rejected' | 'error'
    applied_suggestion INTEGER,       -- 適用された提案のインデックス（0-based）
    error_message TEXT,
    doc_updated_at TIMESTAMPTZ,       -- 提案時のドキュメント updated_at（更新検知用）
    created_at TIMESTAMPTZ DEFAULT now()
);

-- 処理対象の判定:
-- 1. 未整理（folder_id IS NULL）かつ proposals が存在しない → 新規対象
-- 2. 未整理かつ proposals.doc_updated_at < documents.updated_at → 再提案対象
-- 3. 未整理かつ proposals.status = 'rejected' → スキップ（ユーザーが却下済み）

-- 自動モード用のログ
CREATE TABLE organizer_log (
    id BIGSERIAL PRIMARY KEY,
    document_id UUID NOT NULL REFERENCES documents(id),
    document_title TEXT NOT NULL,
    action TEXT NOT NULL,             -- 'rename' | 'move' | 'tag' | 'create_folder' | 'create_tag'
    detail JSONB,                     -- {"old_name": "...", "new_name": "..."} 等
    reason TEXT,
    created_at TIMESTAMPTZ DEFAULT now()
);
```

### suggestions JSONB の構造

```json
[
  {
    "filename": "2026年03月15日_A社見積書.pdf",
    "folder": "営業部/見積書",
    "folder_id": "uuid-or-null",
    "folder_description": "見積書・発注書を保管",
    "tags": ["見積書", "A社"],
    "reason": "ファイル内容がA社宛の見積書であるため。日付は文書内の発行日から取得。"
  },
  {
    "filename": "A社_見積書_20260315.pdf",
    "folder": "営業部/A社",
    "folder_id": null,
    "folder_description": "A社関連の文書",
    "tags": ["見積書", "A社"],
    "reason": "取引先別のフォルダ分類。"
  }
]
```

## 管理画面設定

| 設定キー | デフォルト | 説明 |
|---------|----------|------|
| `organizer_enabled` | `false` | AI振分け機能の有効/無効 |

## 実装手順

### Phase 1: データモデル + UI基盤
1. Folder テーブルに `description` 追加
2. `organizer_rules` テーブル作成
3. `organizer_proposals` テーブル作成
4. `organizer_log` テーブル作成
5. フォルダ作成/編集ダイアログに「目的」フィールド追加
6. 管理画面にAI振分け設定タブ（モード切替 + ルール管理）

### Phase 2: エージェント実装
1. 振分け用ツール定義（list_folders, list_tags, get_file_metadata 等）
2. 振分けプロンプト + ルール注入
3. LLM スケジューラ（検索優先の排他制御）
4. 提案モード: 結果を `organizer_proposals` に保存

### Phase 3: 提案UI
1. 未整理ファイル一覧に提案カード表示
2. 「適用」ボタンで名前変更・フォルダ移動・タグ付与
3. 「すべて適用」で最優先案を一括適用
4. 「却下」で提案を非表示

### Phase 4: 自動モード
1. 自動適用ロジック（提案の最優先案を自動実行）
2. 自動モード用のログ表示
